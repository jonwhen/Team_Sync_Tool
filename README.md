# Team Sync Tool

Team Sync Tool is a real-time collaborative meeting application designed for teams running daily standups, weekly syncs, or any recurring team meeting. It keeps everyone on the same page by providing a shared agenda, live meeting notes, a project board, and more — all accessible from the browser with no accounts required.

The tool promotes active collaboration during meetings by giving every participant the ability to contribute notes, share resources, log decisions, and recognize teammates in real time. When the meeting ends, the team can walk away with a downloadable summary of everything discussed — generated by AI or as a structured template — so nothing falls through the cracks.

## Features

- **Shared Agenda Checklist** — Add agenda items before or during the meeting. Check them off as they are covered so the team can track progress in real time.
- **Collaborative Meeting Notes** — Every participant can write notes simultaneously. Notes appear in a live chat-style feed with author names and timestamps, polled every 5 seconds.
- **Project Board** — A full drag-and-drop kanban board with six columns (To Do, In Progress, In Review, Blocked, Parking Lot, Done). Cards support optional ETA dates and persist across meetings.
- **Meeting Timer** — A 1-hour countdown timer with extend (+15 min) and end controls. The timer turns green when active and pulses red when overtime.
- **Wins & Shoutouts** — A dedicated space for recognizing teammates and celebrating wins during the meeting.
- **Decisions Log** — Capture decisions made during the meeting so the team has a clear record of what was agreed upon.
- **Resources** — Share links and documents relevant to the discussion. Resources display with clickable URLs and attribution.
- **Attendee Tracking** — Users are automatically registered as attendees when they join today's meeting. Attendees appear as pill badges at the top of the page.
- **Meeting History** — A dropdown selector lets the team review meetings from the past 7 days, including all agenda items, notes, and board state from that day.
- **Meeting Summary on End** — When the timer ends, the team is presented with three options:
  - **AI Summary** — Uses the Claude API to generate a narrative summary of the entire meeting (requires an Anthropic API key).
  - **Template Summary** — Formats all meeting data into a clean, structured plain-text file with no external dependencies.
  - **Just End** — Resets the timer without generating a summary.

## Getting Started

### Prerequisites

- Python 3
- Node.js and npm
- Git

### Installation

1. Clone the repository:

```bash
git clone https://github.com/your-username/Team_Sync_Tool.git
cd Team_Sync_Tool
```

2. Install all dependencies:

```bash
make install
```

This runs `pip3 install -r requirements.txt` (Flask, Anthropic SDK, python-dotenv) and `npm install` (Hono server).

3. (Optional) Set up the Anthropic API key for AI-generated summaries:

Open the `.env` file in the project root and replace the placeholder with your key:

```
ANTHROPIC_API_KEY=sk-ant-your-key-here
```

You can get an API key from [console.anthropic.com](https://console.anthropic.com) under Settings > API Keys. This is only required if you want to use the AI Summary option when ending a meeting. The Template Summary and Just End options work without it.

4. Start the application:

```bash
make start
```

This launches two servers in parallel:
- **Flask API** on `http://localhost:5000` (backend)
- **Hono server** on `http://localhost:3000` (frontend)

5. Open your browser and go to:

```
http://localhost:3000
```

Enter your name when prompted and you are in the meeting.

## Architecture

The application uses a three-tier architecture: a vanilla JavaScript frontend, a Node.js middleware server, and a Python Flask backend with SQLite for persistence.

### Frontend (Client) — JavaScript, HTML, CSS

The frontend is a single-page application built with vanilla JavaScript, HTML, and CSS (with Tailwind CSS utility classes for rapid layout). There are no build tools or frameworks — the browser loads the files directly.

| File | Description |
|------|-------------|
| `templates/index.html` | The single HTML page for the entire application. Contains the name prompt overlay, end-meeting choice dialog, summary loading overlay, the meeting view (3-column grid with resources, agenda/shoutouts, and notes), and the project board view (6-column kanban grid). Tab buttons switch between the meeting and board views. All dynamic content is rendered by JavaScript into placeholder elements. |
| `static/app.js` | All client-side logic. Handles user name collection and localStorage persistence, meeting creation and loading, agenda item CRUD with checkbox toggling, live notes with append-only rendering and 5-second polling, drag-and-drop project board with positional reordering, shoutouts/decisions/resources management, the meeting timer (start, extend, end), the end-meeting summary flow with overlay UI, and a consolidated polling loop that keeps all sections in sync across multiple users. |
| `static/style.css` | Custom styles layered on top of Tailwind utilities. Defines the color-coded section system (blue for agenda, green for notes, amber for shoutouts, purple for resources, teal for attendees), the sticky header bar, timer state colors and animations, board card drag-and-drop visual feedback, overlay backdrops with blur effects, button hover/active micro-interactions, custom thin scrollbars, and responsive breakpoints at 1100px and 720px for tablet and mobile layouts. |
| `static/logo.png` | The Team Sync logo displayed in the welcome overlay and the app header. |

### Middleware (Server) — Node.js

| File | Description |
|------|-------------|
| `server.js` | A lightweight HTTP server built with the Hono framework and `@hono/node-server`. Serves static files from the `static/` directory and returns `templates/index.html` on the root route. Runs on port 3000. Its only job is to serve the frontend assets — all data operations go directly from the browser to the Flask API. |
| `package.json` | Declares the two Node dependencies: `hono` (web framework) and `@hono/node-server` (Node adapter). |

### Backend (API + Database) — Python

| File | Description |
|------|-------------|
| `backend/app.py` | The Flask REST API. Defines 30+ endpoints covering CRUD operations for meetings, agenda items, notes, board cards, action items, attendees, decisions, resources, and shoutouts. Also provides two summary generation endpoints: one that calls the Claude API for AI summarization, and one that builds a template-based plain-text summary. Loads environment variables from `.env` via python-dotenv and enables CORS for all origins. |
| `backend/database.py` | The SQLite data access layer. Initializes 9 tables (meetings, agenda_items, notes, board_cards, action_items, attendees, decisions, resources, shoutouts) on startup. Provides functions for all database operations including the `get_meeting_with_items()` function that fetches a complete meeting with all related data in a single call. Enforces foreign keys and uses a unique constraint on attendees to prevent duplicate registrations. |
| `backend/summarize.py` | The AI summarization module. Takes a full meeting data object, builds a structured prompt with all sections (attendees, resources, agenda with covered/not-covered status, raw notes, decisions, shoutouts, board cards by column), and sends it to Claude Sonnet via the Anthropic SDK. Returns a plain-text summary. Raises a clear error if the API key is not set. |
| `requirements.txt` | Python dependencies: `flask` (API framework), `anthropic` (Claude SDK), `python-dotenv` (environment variable loading). |

### Supporting Files

| File | Description |
|------|-------------|
| `Makefile` | Build automation with two targets: `install` (installs Python and Node dependencies) and `start` (launches both servers in parallel). |
| `.env` | Stores the `ANTHROPIC_API_KEY` environment variable. Git-ignored to keep credentials out of version control. |
| `.gitignore` | Standard Python gitignore with additions for SQLite databases, Node modules, `.env`, and OS files. |
| `PROMPT_LOG.md` | A development log tracking each prompt used during the AI-assisted development process and the changes that resulted from each one. |

## Design and Feature Decisions

### Why a 3-column layout?

The meeting view uses a 3-column grid (resources | agenda + shoutouts | notes) to let participants see everything at once without scrolling between sections. Resources sit on the left as reference material, the agenda and shoutouts occupy the center as the primary focus, and notes live on the right as a running conversation. This layout mirrors how meetings naturally flow: check the resources, work through the agenda, and capture notes along the way.

### Why a name prompt instead of full authentication?

Team Sync is designed for teams that are already in a meeting together — they do not need login credentials to collaborate. A simple name prompt stored in localStorage keeps the barrier to entry as low as possible. Anyone can open the URL and join immediately.

### Why 5-second polling instead of WebSockets?

Polling keeps the architecture simple with no additional infrastructure. Meeting notes, board changes, and attendees update every 5 seconds, which is fast enough for a meeting context where people are talking between updates. This avoids the complexity of WebSocket connection management while keeping the experience collaborative.

### Why three summary options on meeting end?

Not every team has an Anthropic API key, and not every meeting needs an AI-generated summary. The three options (AI Summary, Template Summary, Just End) ensure the end-meeting flow works for everyone. Teams that want a polished narrative use AI Summary. Teams that want a quick export use Template Summary. Teams that just want to stop the timer use Just End.

### Why a kanban board inside a meeting tool?

Meetings often produce action items and project updates that need to be tracked visually. Instead of requiring a separate tool like Trello or Jira, the built-in project board lets teams move cards during the meeting as they discuss status. The board persists across meetings so it carries forward from week to week.

### Why does the timer pulse red on overtime?

Meetings that run over time are a common frustration. The red pulsing animation creates gentle urgency without being disruptive — it signals to the group that they are past the scheduled time so they can decide whether to extend or wrap up.

### Why 7-day meeting history?

A weekly retention window covers most recurring meeting cadences (daily standups, weekly syncs) while keeping the meeting selector dropdown manageable. Older meetings still exist in the database and could be accessed by extending the query window if needed.

### Why SQLite?

SQLite requires zero setup — no database server, no connection strings, no credentials. The database file is created automatically on first run. For a team meeting tool where data is local and write volume is low, SQLite provides everything needed without adding infrastructure complexity.
